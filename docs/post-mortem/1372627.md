# Post-mortem: popover mis-positioned inside auro-drawer and auro-dialog

- **Ticket:** AB#1372627
- **GitHub issue:** [auro-popover#130](https://github.com/AlaskaAirlines/auro-popover/issues/130)
- **Affected components:** `@aurodesignsystem/auro-popover`, surfaced via `@aurodesignsystem/auro-drawer` and `@aurodesignsystem/auro-dialog`
- **Resolution:** native HTML Popover API + paired Popper.js v2 adjustments

## Problem

When `<auro-popover>` was nested inside the content of either wrapper:

- **Inside `<auro-drawer>`:** the bubble (and its caret) sat ~122px to the right of the "Hover on me" trigger. Vertical placement was correct; horizontal was off by a constant offset that matched the drawer's left edge.
- **Inside `<auro-dialog>`:** the bubble was correctly centered on the trigger but visually clipped at the dialog's left edge — the bubble extended past `.dialog-content`'s boundary and the dialog's nested `overflow: hidden` rules cut it off.

Both bugs reproduced reliably in the wrapper components' `popover-and-dropdown.html` API examples.

## Investigation

### Attempt 1 — `strategy: "fixed"` alone

**Hypothesis.** Popper.js v2 mis-projects coordinates when the popper's offsetParent chain crosses a transformed ancestor (drawer's `.wrapper` uses `transform: translate3d(...)` for its slide-in animation; dialog's `.bib` and `dialog.container` use `transform: translateZ(0px)`). Popper's own documentation recommends `strategy: "fixed"` for popovers inside transformed or fixed-positioned containers.

**Outcome.** The dist was rebuilt with the strategy option set, deployed into `auro-drawer/node_modules`, and verified to contain `strategy:"fixed"`. The drawer bug reproduced identically — same drift, same caret offset.

**Why it didn't work.** `strategy: "fixed"` only changes which CSS positioning method Popper writes (`position: fixed` vs. `position: absolute`). It does not change how Popper detects the containing block. Both strategies share the same broken `getContainingBlock` walk. The dialog clipping was also untouched, because clipping is an overflow concern, not a positioning concern.

### Attempt 2 — `altAxis: true` + `boundary: "clippingParents"`

**Hypothesis.** For the dialog case specifically, the `preventOverflow` modifier was only running `mainAxis: true`. For `placement: top` Popper's "main axis" is the centering axis (X), but the clamping wasn't shifting the bubble away from the dialog's left edge because the boundary defaulted to `document.body`. Adding `altAxis: true` and switching the boundary default to Popper's `'clippingParents'` sentinel would let Popper auto-shift the bubble to fit inside the nearest clipping ancestor.

**Outcome.** Set aside before deployment. The fix would have solved dialog clipping by shifting the bubble rather than letting it overflow, but the user surfaced the dialog issue as a separate concern from the ticket (which was scoped to drawer). More importantly, this would not have addressed the drawer's positioning bug, which is not an overflow problem.

### Diagnostic data

Captured by inspecting the live drawer in DevTools:

```js
trigger.getBoundingClientRect()   // left=201.4, width=91.9, center=247.4
.wrapper.getBoundingClientRect()  // left=137.4, width=1236.6, top=0

// popover computed styles
position: fixed
left: 0px
top: 734px
right: 756.594px
bottom: 0px
transform: matrix(1, 0, 0, 1, 16, -682)   // translateX=16, translateY=-682
max-width: 432px
width: 432px
```

Working the numbers:

- Browser visual position: `wrapper.left + style.left + transform.x = 137 + 0 + 16 = 153`
- Bubble width 432 → bubble center at viewport x = 153 + 216 = 369
- Expected (centered on trigger): 247
- **Drift = +122px**, matching the visual report exactly.

Two pieces of information were now ground-truth: the bubble was positioned with `left: 0` interpreted against `.wrapper`'s left edge (137px), and Popper had written `transform: translateX(16px)` as if origin were viewport (0px).

## Root cause

Popper.js v2's `getContainingBlock` walk reads (de-minified) approximately:

```js
var i = C(e);                                              // step 1: shadow-aware
for (x(i) && (i = i.host); b(i) && [...].indexOf(v(i)) < 0;) {
  if ("none" !== s.transform || ...) return i;             // check for transform/perspective/etc.
  i = i.parentNode;                                        // step N: NOT shadow-aware
}
return null;                                               // fallback → viewport
```

`C(e)` is shadow-aware (it uses `assignedSlot` and `host`) and is used exactly once at the start. Every subsequent iteration uses plain `parentNode`, which:

- Does NOT follow `assignedSlot` from a slotted element into its host's shadow
- Does NOT cross from a shadow root's contents into the shadow host (unless wrapped explicitly)

For our drawer DOM after auro-drawer's `updated()` lifecycle relocates the user's slotted children into `<auro-drawer-content>`:

```
popover_div (auro-popover shadow root)
  → C() → auro-popover host
  → parentNode → <div slot="content">
  → parentNode → <auro-drawer-content>     (display: contents, no transform)
  → parentNode → <auro-drawer>             (no transform)
  → parentNode → <body>                    (exits)
returns null → falls back to window/viewport
```

`.wrapper` — the transformed element that actually establishes the containing block — lives inside `<auro-drawer-content>`'s **shadow**. The flat-tree walk reaches it via the slot-assignment chain; Popper's `parentNode` walk does not.

So:

- **Browser** (walking the flat tree per CSS spec) identifies `.wrapper` as the containing block, interprets `left: 0` as `.wrapper.left = 137px`, then applies `transform: translateX(16px)` → final visual x = 153.
- **Popper** (walking `parentNode`) doesn't find any transform ancestor, computes coordinates assuming origin = viewport (x = 0), and writes a transform that would have produced the correct centering IF origin really were 0.

The mismatch is exactly the offset between what Popper assumed (viewport) and what the browser used (`.wrapper`), minus a small adjustment from `preventOverflow`'s 16px boundary padding. Hence the constant ~122px drift.

The dialog clipping is a related but distinct manifestation of the same architectural issue: nested `overflow: hidden` rules on `.bib`, `dialog.container`, and `overflow: auto` on `.dialog-content` clip any descendant that overflows them. Popper's positioning code has no way to "escape" those constraints from within the dialog's DOM subtree.

## The fix

A coordinated set of three changes in `auro-popover`:

### 1. Promote the bubble to the top layer

Add `popover="manual"` to the bubble in the render template, and call `showPopover()` / `hidePopover()` from `Popover.show()` / `Popover.hide()` (guarded by `isConnected`, `typeof showPopover === "function"`, and `:popover-open` checks).

Per the CSS spec, top-layer items have their containing block set to a viewport-sized box regardless of any ancestor's `transform`, `perspective`, `filter`, `overflow`, or `clip-path`. The browser-side coordinate system becomes the viewport unconditionally.

### 2. Tell Popper to write viewport-relative coordinates

Set `strategy: "fixed"` on the Popper instance so the bubble gets `position: fixed` inline, matching the top-layer browser semantics.

### 3. Override Popper's reference rect when the bubble is in the top layer

A custom `beforeRead` modifier rewrites `state.rects.reference` to viewport-relative coordinates whenever `bubble.matches(":popover-open")` is true:

```js
{
  name: "viewportReferenceForTopLayer",
  phase: "beforeRead",
  enabled: true,
  fn: ({ state }) => {
    const popper = state.elements.popper;
    if (typeof popper.matches !== "function" || !popper.matches(":popover-open")) {
      return;
    }
    const r = state.elements.reference.getBoundingClientRect();
    state.rects.reference = { x: r.left, y: r.top, width: r.width, height: r.height };
  },
}
```

This piece was discovered during testing: with only steps 1 and 2, a regression test that placed the popover inside a transformed `<div>` in light DOM (a case where Popper's `parentNode` walk *succeeds*) failed by ~8px. The walk found the transformed ancestor and computed the reference rect in those coords, while the bubble — now in the top layer — was positioned against the viewport. Without the override, the two coordinate systems disagreed by the offsetParent's viewport offset.

In the drawer scenario the walk fails (shadow boundaries), so steps 1 and 2 alone would have been enough. The modifier exists for the cases where Popper *can* see a transformed ancestor and would otherwise mis-align by its offset.

### 4. Neutralize UA stylesheet defaults

The browser's UA stylesheet applies these to any element with the `popover` attribute (via `:where([popover])`, specificity 0,0,0):

```css
position: fixed; inset: 0; width: fit-content; height: fit-content;
margin: auto; border: solid; padding: 0.25em; overflow: hidden;
color: CanvasText; background-color: Canvas;
```

Most of these were already overridden by author CSS or by Popper's inline styles (`position`, `inset`, `margin`, `display`, `padding` via `util_insetLg`, `background-color` via `color.scss`). Two remained:

- `border: solid` — would add a visible UA-default border.
- `overflow: hidden` — would clip the arrow (which protrudes 10px below the bubble via `bottom: -10px`).

Added `border: 0;` and `overflow: visible;` to `.popover`.

## Why this works

| Concern | Before | After |
|---|---|---|
| Browser-side containing block | `.wrapper` (transformed ancestor) | Viewport (top-layer rule) |
| Popper-computed reference rect | offsetParent-relative (varies) | Viewport-relative (forced by modifier) |
| Popper-written CSS position | `position: absolute`, offset-parent-relative | `position: fixed`, viewport-relative |
| Coordinate-system agreement | ❌ disagrees by ancestor offset | ✅ both viewport |
| Bubble clipped by ancestor `overflow: hidden` | ✅ yes (dialog) | ❌ no (top-layer escapes clipping) |

The same change resolves both the drawer drift (a coordinate-system disagreement) and the dialog clipping (an overflow-escape problem) because top-layer rendering decouples the bubble from ancestor layout in both senses.

## Browser support and fallback

- Chrome 114 (May 2023)
- Safari 17 (September 2023)
- Firefox 125 (April 2024)

On older browsers, `showPopover` is `undefined`, the feature check skips it, the `:popover-open` test in the custom modifier returns `false`, and Popper falls back to its default behavior. The drawer/dialog bugs would re-appear on older browsers, but no new regression is introduced. Given the > 1 year window since all three major engines shipped, the fallback path is exercised only on aging installs.

## Verification

- **Unit tests:** four new tests in `test/auro-popover.test.js` under the `top-layer promotion` describe block — verify the `popover="manual"` attribute is present, the bubble enters and leaves `:popover-open` in sync with mouse events, and the centering matches within 2px when the popover lives inside a transformed light-DOM ancestor (regression for the asymmetry uncovered during this work). Full suite: 68 passed.
- **Manual:** both wrapper API examples — `auro-drawer/apiExamples/popover-and-dropdown.html` and `auro-dialog/apiExamples/popover-and-dropdown.html` — visually confirmed correct centering and no clipping.

## Lessons

- **Trust DevTools numbers earlier.** The first round of fixes was driven by a plausible-sounding theory ("strategy: 'fixed' fixes transformed-ancestor positioning per Popper docs") rather than by ground-truth data from the broken page. Getting `getBoundingClientRect` values and computed styles from the live drawer was what made the math fall into place — and that should have been step one.

- **Web Test Runner fixtures can hide shadow-DOM-specific bugs.** A straightforward fixture with a transformed `<div>` wrapping `<auro-popover>` does NOT reproduce the drawer bug, because the transformed ancestor is reachable via `parentNode`. The bug only manifests when the transformed ancestor lives inside *another* component's shadow. Tests that simulate the wrapper context end-to-end are more valuable than unit-style tests for component-composition issues like this.

- **Popper.js v2's containing-block detection has a known gap around shadow DOM.** When choosing between Popper v2, Floating UI v1+, and the native CSS Anchor Positioning spec for future positioning work, the shadow-traversal behavior is worth checking explicitly — Floating UI v1 fixed several of these. This fix works around the issue without an engine swap, but a future migration off Popper v2 would let several of the workarounds (the custom modifier in particular) be deleted.
